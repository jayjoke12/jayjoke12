# Jay

I build Minecraft systems — Bedrock, Java, Spigot — and the occasional interactive web project when I want to work outside the game.

Most of what I make is about changing how players interact with a world, not adding content to it. There’s a difference.

-----

## What I build

- Multiplayer systems (shared inventories, synced states, player interaction logic)
- Gameplay modifiers (cursed blocks, damage indicators, rule-changing mechanics)
- Mods and plugins (Java Edition + Spigot)
- Small interactive web projects tied to real actions or events

-----

## Projects

### InvShare (MCBE)

Shared inventory system using the Bedrock Script API. All linked players share one inventory state — one player picks something up, everyone has it.

-----

### CursedBlocks (Spigot 1.21.4)

Every few minutes, a random block type becomes lethal. The game keeps shifting underneath you.

-----

### Totem Indicator (Java mod)

Visual damage tracking for combat. The HUD doesn’t tell you much about pressure in the moment — this fills that gap.

-----

### Head Tracker (Java / Fabric + Python)

A mod that maps real head movement to in-game camera input using computer vision. Turn your head left, the camera follows — same feel as a joystick, no controller needed.

-----

### Valentine’s Day Card (Web)

An interactive page that turns a “yes/no” into a small event. Animation effects, Discord webhook notifications. Silly, but it worked.

-----

## The common thread

I keep coming back to the same problem: how do you make a system that changes how people play, not just what they have access to? Decision pressure, shifting states, feedback loops — and lately, rethinking the input itself. That’s most of what I think about.
